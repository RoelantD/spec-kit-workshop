### The Spec-Driven Development Workflow Review

#### Phase 1: Specify
- **Input**: Project idea, user needs
- **Process**: Use `/specify` to generate specification
- **Output**: Detailed spec focused on user journeys and outcomes
- **Key question**: "What should we build and why?"

#### Phase 2: Plan
- **Input**: Completed specification
- **Process**: Use `/plan` to generate technical plan
- **Output**: Technical architecture, stack, and implementation strategy
- **Key question**: "How should we build it?"

#### Phase 3: Tasks
- **Input**: Specification and plan
- **Process**: Use `/tasks` to break down into implementable pieces
- **Output**: Detailed task list with acceptance criteria
- **Key question**: "What specific work needs to be done?"

#### Phase 4: Implement
- **Input**: Task list
- **Process**: AI agent implements tasks one by one
- **Output**: Working, tested code that fulfills the specification
- **Key question**: "Does this implementation meet our requirements?"

### Practical Applications

#### When to Use SDD
- **Greenfield projects**: Starting from scratch
- **Feature additions**: Adding to existing systems
- **Legacy modernization**: Rebuilding old systems
- **Learning new technologies**: Structured exploration
- **Team collaboration**: Shared understanding

#### When SDD Might Be Overkill
- **Trivial changes**: One-line bug fixes
- **Prototypes**: Quick throwaway experiments
- **Well-understood patterns**: Repetitive boilerplate

### Common Questions

#### "Do I always need all four phases?"
- **Short answer**: For best results, yes
- **Nuance**: Small changes might not need full process
- **Recommendation**: When in doubt, use all phases

#### "Can I use SDD without AI?"
- **Yes**: SDD principles work without AI
- **But**: AI makes specs executable, unlocking their full power
- **Alternative**: Use specs for human team coordination

#### "What if my organization doesn't use GitHub Copilot?"
- **No problem**: Spec Kit works with multiple AI agents
- **Supported**: GitHub Copilot, Claude Code, Gemini CLI
- **Extensible**: Can adapt prompts for other agents

#### "How do I convince my team to try SDD?"
- **Start small**: Pilot on one project
- **Show results**: Demonstrate quality improvements
- **Address concerns**: Listen to objections, adjust approach
- **Provide training**: Help team learn the process

#### "Is SDD only for new projects?"
- **No**: Works for existing codebases too
- **Legacy systems**: Great for modernization
- **Feature additions**: Excellent for extending systems
- **Refactoring**: Helps plan and execute improvements
